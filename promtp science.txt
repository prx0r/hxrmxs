Thesis: The BORIS Protocol and the Science of a Self-Optimizing Psycholinguistic Engine
Abstract
This document outlines the definitive architecture for a recursively self-improving prompt engineering pipeline, the core of the Consciousness Archaeology Engine (CAE). The system is built upon the BORIS (Bayesian Optimized Recursive Intelligence System) protocol. The primary innovation is a departure from treating prompts as static, human-crafted strings. Instead, prompts are treated as dynamic, parameter-driven "specifications of intent" that can be algorithmically generated, validated, and optimized. By formalizing a "prompt science" based on established linguistic and psychological principles, we create a system that learns the art of communication itself. It uses a Genesis process for novel concept induction, a Spark process for creative exploration, and a Logos dual-learning system to build a Bayesian preference model of conversational effectiveness. This thesis provides the complete architectural and algorithmic blueprint for this engine, detailing how prompt quality is ensured, how it improves over time, and how it remains aligned with the system's strategic goals.


1.0 The Foundation: The "Pragmatic Prompt Framework"
To create a science of prompts, we must first establish a language to describe them. The Pragmatic Prompt Framework is this language, a multi-dimensional system for defining the precise nature and intent of any psychological probe. This framework is implemented in the codebase as a series of Python Enum and dataclass objects, ensuring all specifications are explicit and type-safe.

1.1 Illocutionary Act (The Prompt's Function): Defines what the prompt is doing. Based on Searle's taxonomy, it replaces ambiguous goals with formal functions.

DIRECTIVE: Instructs the user to perform a cognitive act (e.g., "Describe a time..."). This is the workhorse of our assessment.
EXPRESSIVE: Conveys the AI's state to build rapport (e.g., "That sounds like a challenging situation.").
ASSERTIVE: States a belief to test a hypothesis (e.g., "It seems you place a high value on loyalty.").
1.2 Cognitive Demand (The Prompt's Complexity): Defines the level of cognitive effort required from the user, moving beyond a simple 1-10 scale to a functional hierarchy.

REPRODUCTION (Level 1): Recall of information ("When did that happen?").
INTERPRETATION (Level 2): Explaining or summarizing ("What do you think that experience meant?").
ANALYSIS (Level 3): Comparing or breaking down concepts ("How did your reaction then differ from how you'd react now?").
APPLICATION (Level 4): Applying knowledge to a new scenario ("Given a similar situation, what would you do next?").
1.3 Social Valence (The Prompt's Relational Context): Defines the social and emotional texture of the prompt.

Formality: The linguistic register (e.g., formal vs. informal).
Politeness: The use of strategies to maintain social harmony.
Emotional Framing: Whether the topic is framed positively, negatively, or neutrally.
Every prompt generated or analyzed by the system is described by a PromptSpecification object containing these precise parameters. This is the bedrock of our science.

2.0 The Quality Assurance Pipeline: How We Ensure Good Prompts
A "good prompt" is one that is strategically sound, computationally efficient to discover, and semantically faithful to its validated specification. We ensure this through a multi-stage pipeline that combines predictive modeling, recursive refinement, and validation.

2.1 Stage 1: Predictive Filtering (The P3 Shortcut): To avoid the immense computational cost of refining every possible prompt, the Spark module first generates a large batch (50+) of candidate PromptSpecification objects. These lightweight blueprints are fed into the P3 (Prompt Performance Predictor), a trained LightGBM model that returns an instant prediction of each specification's likely effectiveness. The BORIS orchestrator then prunes this set, selecting only the top 2-3 candidates for further processing.

2.2 Stage 2: Recursive Refinement (Genesis): The highest-scoring PromptSpecification is passed to the Genesis module. This module executes an intra-turn generate -> evaluate -> refine loop. It constructs a meta-prompt from the spec, tasks Mamba-2 with generating an improved version of the spec, evaluates the output against an internal rubric, and repeats this process until the specification's quality score meets a predefined convergence target. This ensures the blueprint is internally optimized before any user-facing content is created.

2.3 Stage 3: Pre-Generation Validation (The PQA Crew): The final, refined PromptSpecification from Genesis is sent to the Internal Review Board. This is a fast, specialized ThinkTank crew that validates the blueprint itself. It checks that the target_concept is valid according to the Psycho-Analytic Knowledge Graph (PKG) and that the strategic combination of pragmatic parameters is coherent and safe.

2.4 Stage 4: Post-Generation Validation (Semantic Drift Check): After the validated specification is sent to Mamba-2 to generate the final text, a lightweight, automated function performs a final check. It generates an embedding for the Mamba output and compares its cosine similarity to the original target_concept embedding from the PKG. If the similarity is too low, it signals a "semantic drift" or hallucination. The system discards the output, uses a safe fallback, and logs the failure event for offline analysis.

3.0 The Engine of Improvement: The Dual Logos Learning Loop
This is how the system's "prompt science" improves over time. The entire architecture is a flywheel designed to generate and learn from a unique dataset.

3.1 The Golden Datapoint: After every turn, the Logos_Learner module calculates the true effectiveness of the deployed prompt, using metrics like AIG (Actualized Information Gain - how much the prompt reduced the system's uncertainty) and RQE (Response Quality Elicitation - the lexical richness and complexity of the user's response). This creates our fundamental unit of learning: a (PromptSpecification, TrueEffectivenessScore) pair.

3.2 The Dual Logos Architecture: This golden datapoint is used to update two models simultaneously, solving the N=1 to N=Many scaling problem.

PersonalLogos: The data updates a user-specific Bayesian model, allowing the system to learn an individual's unique conversational preferences (e.g., "This user responds well to prompts with high CognitiveDemand but low Formality"). The system's recommendations are a blend of global and personal models, with the weighting shifting towards the personal model as it learns more about the user.
GlobalLogos: The anonymized data is added to a massive central database. This database trains the system on the universal principles of what makes a prompt effective across the entire user population.
3.3 The Meta-Learning Flywheel: This is the highest level of improvement.

Conversations generate thousands of (Spec, Score) datapoints, which enrich the GlobalLogos database.
Periodically, this entire growing dataset is used to re-train the P3 Predictor model offline.
A newly retrained, "smarter" P3 model makes more accurate predictions at the start of the real-time loop.
Better predictions mean the system wastes less time on poor candidates and focuses its powerful Genesis refinement process on specifications with higher potential.
This leads to the generation of more effective prompts for the user.
More effective prompts elicit higher-quality data, which in turn creates better training examples for the next generation of the P3 model.
This flywheel ensures the system's core "prompt science" gets progressively more intelligent and efficient with every conversation it has.

4.0 Coherence with System Goals: The BORIS Orchestrator
The prompt science does not operate in a vacuum. Its coherence with the overall system goals is maintained by the BORIS Orchestrator, the top-level class that functions as the system's "consciousness."

Top-Down Strategic Direction: At the beginning of every turn, the orchestrator analyzes the complete ConversationState, including the user's current PerceivedSelfGraph, the uncertainty levels for each personality trait, and any detected dissonances.
Goal-Oriented Tasking: Based on this analysis, the orchestrator sets the explicit target_concept for the entire prompt generation pipeline. It dictates the "why" before the sub-modules figure out the "what" and "how." For instance, if uncertainty about 'Conscientiousness' is highest, it will command the pipeline to generate a prompt specifically targeting that trait. This top-down, goal-oriented direction ensures that no matter how creative the prompt generation becomes, it is always in service of the primary mission: to efficiently and accurately model the user's psyche.

By treating prompts as scientifically-defined, parameter-driven specifications, we create a system that can algorithmically reason about and optimize its own communication. 3. Defining the "TrueEffectivenessScore": A Quantifiable Approach
You have rightly pointed out that calculating the TrueEffectivenessScore is a non-trivial prerequisite for the entire learning loop. Here is a concrete, technical proposal for calculating its two main components, AIG and RQE.

AIG (Actualized Information Gain): A Measure of Uncertainty Reduction

Representation: The system's BeliefState about the user is not just a vector of scores. It's a probability distribution, likely a multivariate Gaussian distribution over the personality traits, defined by a mean vector (μ) and a covariance matrix (Σ).
Quantifying Uncertainty: The system's total uncertainty can be quantified as the entropy of this distribution. A simple and effective proxy for this is the trace (the sum of the diagonal elements) of the covariance matrix: Uncertainty = tr(Σ). A larger trace means higher variance and thus higher uncertainty.
Calculating AIG: After a user's response is processed, the BeliefState is updated via a Bayesian update rule, resulting in a new covariance matrix, Σ

AIG = (tr(Σ_old) - tr(Σ_new)) / tr(Σ_old)
This provides a clean, quantifiable measure of how much the system "learned" from a given turn.
RQE (Response Quality Elicitation): A Measure of Linguistic Richness

This is a composite score calculated from the user's raw text response, designed to measure its depth and complexity.
It is a weighted average of several NLP metrics:
Response Length: Simple word count, normalized.
Lexical Diversity: The Type-Token Ratio (unique words / total words). A higher ratio indicates more varied vocabulary.
Syntactic Complexity: The depth of the sentence's dependency parse tree, as determined by a library like spaCy. Deeper trees imply more complex sentence structures.
Semantic Richness: We generate embeddings for every word in the user's response and calculate the volume of the convex hull of these embeddings in the vector space. A larger volume suggests a wider range of concepts was discussed.
The final TrueEffectivenessScore is a weighted average of these components, e.g., score = (0.6 * AIG) + (0.4 * RQE). These weights themselves are system parameters that can be tuned over time.

1.2. Architecting the Two-Speed System: Real-Time vs. Asynchronous
As you directed, the ThinkTank crews are not in the real-time loop. The BORIS Orchestrator manages a fast, real-time cycle and dispatches longer-running tasks to the asynchronous "Builder" crews.

Real-Time Loop (Low Latency): Handles the immediate user conversation. Its goal is to generate the best possible next prompt in under a second.
Asynchronous Loop (High Latency, Deep Work): Handles research, content creation, and meta-learning. This is where the ThinkTank crews operate, triggered by events but running independently in the background (e.g., via Celery or BackgroundTasks in FastAPI).
1.3. The Real-Time Prompt Generation Cycle: Spark & The P3 Predictor
This is the start of the "prompt science" feedback loop, as detailed in the roadmap.

Candidate Generation (Spark): At the start of a turn, the Spark_Explorer module generates 50+ candidate PromptSpecification objects. It does this by taking a target concept (e.g., "Agreeableness") and creating numerous permutations of the PragmaticPromptVector.
Computational Shortcut (P3): These 50+ lightweight PromptSpecification objects (not full text prompts) are vectorized and fed as a batch into the P3 (Prompt Performance Predictor). This is a pre-trained, lightweight LightGBM model that instantly predicts the effectiveness score for each candidate specification.
Intelligent Pruning: The BORIS Orchestrator receives the 50+ predicted scores and selects only the top 2-3 most promising candidates, discarding the rest. This prevents wasting computational resources on low-potential ideas.
1.4. The Recursive Refinement Cycle: Genesis & The PQA Crew
This cycle takes the best blueprints from Spark/P3 and polishes them into masterworks.

Recursive Refinement (Genesis): The highest-scoring PromptSpecification from the previous step is passed to the Genesis_Innovator. This module runs the generate -> evaluate -> refine loop described in prompt engineering.txt. It uses Mamba-2 to generate a meta-prompt based on the spec, evaluates its own work against an internal rubric, and refines the meta-prompt over several cycles until it meets a quality threshold.
Specification Validation (PQA Crew): The final, refined PromptSpecification is now sent to the Internal Review Board. This is the first place a ThinkTank crew is used in the real-time loop, but it's a fast, specialized one. Its agents validate the blueprint against the PKG for psychological and strategic soundness.

Final Generation (Mamba-2): This is your crucial correction incorporated. The validated specification is then used to construct the final, precise meta-prompt that is sent to Mamba-2 to generate the user-facing text.
1.5. The Logos Dual-Learning Loop: How the "Science" Improves
This is the most important part of the architecture. This is how the system's core value—its knowledge of prompt engineering—grows over time.

Effectiveness Measurement: After the user responds to the prompt, the Logos_Learner module calculates the true effectiveness of the PromptSpecification that was used. It uses metrics like AIG (Actualized Information Gain) and RQE (Response Quality Elicitation).
The Golden Datapoint: The system now has a complete training example: (PromptSpecification, TrueEffectivenessScore).
Dual-Model Update: This datapoint is used to update two distinct models via Bayesian inference:
PersonalLogos: The data updates a small model specific to that user, learning their individual conversational preferences. This ensures the experience becomes more personalized over time.
GlobalLogos: The anonymized data point is added to a massive central database, contributing to the system's universal understanding of psycholinguistics.
Closing the Meta-Learning Loop: Periodically, the entire, ever-growing GlobalLogos dataset is used to re-train the P3 Predictor model offline. This is the critical step: as the system conducts more conversations, its ability to predict which prompts will be effective gets progressively smarter. This makes the real-time "computational shortcut" more accurate, allowing the system to focus its creative energy more efficiently.
